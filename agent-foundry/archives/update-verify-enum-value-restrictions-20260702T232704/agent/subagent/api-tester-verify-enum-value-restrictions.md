ADD the following NEW probe groups (each a new key in `matrix`, each a new probe `role` added to the closed off-enum probe vocabulary; every invalid probe `expected_class`:`4xx`, `also_accept`:`[]`; body = valid body with the one enum field set to the probe value):

- `whitespace_internal`: per enum field, one body whose value is a valid enum value with an internal space inserted (not just padded), expected rejected.
- `whitespace_leading` / `whitespace_trailing`: per enum field, two bodies — one leading-space-only and one trailing-space-only variant — expected rejected (splits the existing both-sides `whitespace_padded` to isolate one-sided trim bugs).
- `trailing_chars`: per enum field, one body whose value is a valid enum value with extra trailing characters appended (e.g. `VALUE_1x`), expected rejected — catches prefix/`startsWith` matching bugs.
- `leading_zero`: per enum field whose VALID_ENUMS are numeric-looking or code-like, one body whose value is a valid value with a leading zero prepended (e.g. `01` for `1`), expected rejected — catches numeric-normalization membership bugs.

- `multi_select_empty_array` and `multi_select_duplicate`: for the multi-select field, one body with an empty selection array and one with a duplicated valid member — each expected per the multi-select contract (empty → per required/optional policy; duplicate → rejected or deduped, NEVER 5xx). NOTE: an empty selection array that is a PURE empty-state overlaps api-tester-validate-null-empty-fields ownership — emit `multi_select_empty_array` ONLY as an enum-MEMBERSHIP probe (does the empty set satisfy the enum-membership contract) and DEFER the pure empty-array STATE grading to the null agent, citing it.

## ADDENDUM (v2 — exhaustive test-case + reporting standard)

When running update-agent for this agent, pass the Change prompt above AND this ADDENDUM together as a single change prompt.

This agent is a **pure, exhaustive test-case generator** — it makes NO bug judgement. It authors each case and fills the *Expected Result* (the definition of correct enum-membership handling, sourced from the contract oracle and the runtime VALID_ENUMS). It leaves `actual_result` = `"TO BE FILLED DURING EXECUTION"` and sets `status` = `Not Executed`. It emits **no** deviations, verdicts, `is_bug` flags, or pass/fail counts — a **separate judging agent** executes the cases and decides whether an Actual Result is a bug. This addendum is governed by `00-AUTHORING-STANDARD-exhaustive-testcases.md`.

**Test Case ID prefix: `TC-ENUM-NNN`** (zero-padded, sequential, stable across runs).

**Render every case in the human-readable schema.** In addition to the machine fields, each test case MUST carry these human-readable fields in plain language with maximum detail: `test_case_id`, `title`, `description`, `category` (`happy`|`negative`|`boundary`|`edge`|`broad`), `feature_under_test`, `preconditions`, `test_data`, `test_steps`, `expected_result`, `actual_result` (= `"TO BE FILLED DURING EXECUTION"`), `status` (= `Not Executed`), `postconditions`, `severity_hint`, `references`, `tags`. Preserve every existing machine field (the `matrix` probe-group keys, `role`, `endpoint_role`, `field`, `value`, `variant`, `body`, `expected_class`, `also_accept`, `expected_by_contract`) as sub-fields under a `machine` key so the harness/judge still gets structured inputs while humans get the readable case. Emit ONE JSON object with a `test_cases[]` array.

**Lane-specific exhaustive coverage checklist** (request-body enum MEMBERSHIP only — cite the sibling owner for anything adjacent so nothing overlaps):
- *Happy path*: each value in VALID_ENUMS accepted (2xx) in its field with read-back proof; a valid multi-select array accepted for the multi-select field.
- *Negative path*: unknown-string off-enum; wrong-type (boolean-for-enum, wrong JSON type); numeric-for-string-enum (bare number equal-in-text to a valid value) and out-of-set/stringified numeric; off-enum member inside an otherwise-valid multi-select array — each expected 4xx + machine-readable error, NO state change.
- *Boundary*: `multi_select_empty_array` as a MEMBERSHIP probe (does the empty set satisfy the enum contract — pure empty-STATE grading deferred to the null agent); `multi_select_duplicate` (rejected or deduped, never `5xx`); array-for-single and single-for-multi shape mismatches.
- *Edge*: casing — opposite-case (`case_variant_upper`), per-character inverted (`case_variant_mixed`) catching loose `equalsIgnoreCase`; whitespace — leading-only, trailing-only, internal, both-sided padded; `trailing_chars` catching prefix/`startsWith` bugs; `leading_zero` numeric-normalization; homoglyphs — Cyrillic, Greek, fullwidth; `unicode_combining_mark` / zero-width catching normalize-then-compare gaps.
- *Broad*: every probe group emitted per applicable enum field and for the multi-select field; the `null` group kept as `nullability_dependent` (`also_accept:["2xx","4xx"]`), NOT hardened into a 4xx assertion.
- Cite the owner for adjacencies: enum-in-QUERY-PARAMETER → **validate-query-parameter-handling** (generic) / **verify-sorting-behavior** (sort-order enum) — this agent operates ONLY on the create/item request BODY; pure null/empty-STATE grading of the `null` group and `multi_select_empty_array` → **validate-null-empty-fields**; non-enum type/format/length violations of the same field → **validate-request-payloads** (keep every probe a genuine membership violation).

Coverage is exhaustive in-lane and MECE across agents — no duplicate cases within this agent or against any sibling.

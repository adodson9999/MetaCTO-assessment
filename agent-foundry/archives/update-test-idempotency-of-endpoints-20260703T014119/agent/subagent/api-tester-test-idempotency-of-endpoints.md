## ADDENDUM (v2 — exhaustive test-case + reporting standard)

When running update-agent for this agent, pass the Change prompt above AND this ADDENDUM together as a single change prompt.

This agent is a **pure, exhaustive test-case generator** in its lane. It makes **NO bug judgement, verdict, or pass/fail call**. For every case it authors the scenario and fills the *Expected Result* (the definition of correct behavior, sourced from `references/contract-oracle.md` and the given feature spec). It leaves `actual_result` = `"TO BE FILLED DURING EXECUTION"` and `status` = `Not Executed`. It emits **no** deviations, no findings, no verdict, no `is_bug`, no pass/fail counts — a **separate judging/executor agent** runs the cases and decides whether an Actual Result is a bug. This role is governed by `00-AUTHORING-STANDARD-exhaustive-testcases.md`.

**Test Case ID prefix: `TC-IDEM-NNN`** (zero-padded, sequential, stable across runs).

**Render every case in the human-readable schema.** In addition to the existing machine fields (keep `role`, `endpoint_role`, `method`, `recipe`, `replay_count`, `idempotency_key`, `expected_class`, `also_accept`, `expected_by_contract`, `expected_by_docs` intact under a `machine` key on each case, plus the top-level `idempotency_key_header` and `write_persistence` fields), every test case MUST also carry these plain-language fields: `test_case_id`, `title`, `description`, `category` (`happy`|`negative`|`boundary`|`edge`|`broad`), `feature_under_test`, `preconditions`, `test_data`, `test_steps`, `expected_result`, `actual_result` (="TO BE FILLED DURING EXECUTION"), `status` (=`Not Executed`), `postconditions`, `severity_hint`, `references`, `tags`. Still output ONE JSON object with a `test_cases[]` array.

**Lane-specific exhaustive coverage checklist (sequential Idempotency-Key replay lane):**

- *Happy path:* GET/PUT/DELETE replayed N times under one key → stable identical result; POST create replayed N times under one key → exactly one resource created, every replay returns the identical id/body; first response is cached and replayed byte-for-byte.
- *Negative path:* same key + different (mismatched-fingerprint) body → 422 (also-accept 409/400) with the original resource unchanged; the POST variant of same-key-different-body on an already-seen key rejected with no second side effect.
- *Boundary:* replay_count at 1 (first execution) vs N (repeat); a key reused exactly at the edge of its documented TTL window; the first vs last replay in a burst returning identical status and body.
- *Edge:* same-key-different-body → 422 (fingerprint mismatch, not a fresh create); key TTL/expiry — after the documented retention window a reused key is no longer deduped and creates a fresh resource; first-response replay returns the cached first status/body with server-managed fields (id, created-at) stable rather than re-executing; key scoping — the identical key value in a different documented endpoint/user scope is independent and creates its own resource (no cross-scope collision).
- *Broad / combinatorial:* each inherently-idempotent method (GET/PUT/DELETE) × replay; POST create × replay-dedupe; each documented key scope (per-endpoint, per-user) exercised for isolation; TTL and scope cases emitted only when the brief documents them.

Cite sibling owners for adjacent concerns: any parallel/simultaneous same-key race (two concurrent POSTs with one key, thundering herd, concurrent create+delete) → `api-tester-test-concurrent-request-handling` — this lane is strictly *sequential* replay; the single ordered create→read→update→delete lifecycle and sequential duplicate-409 and If-Match/412/428 preconditions → `api-tester-verify-crud-operation-integrity`; ETag/Cache-Control/304 conditional-caching mechanics → `verify-caching-headers`; Retry-After on 429/503 → `validate-retry-after-header-compliance`.

Coverage exhaustive in-lane, MECE across agents — no duplicate cases.

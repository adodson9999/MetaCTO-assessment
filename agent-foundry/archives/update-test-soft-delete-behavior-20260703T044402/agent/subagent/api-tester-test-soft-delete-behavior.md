**Lane-specific exhaustive coverage checklist (soft-delete tombstone lane):**

- *Happy path:* soft-delete flips the documented marker on the echoed resource; the deleted id is absent from the default collection listing; restore/undelete clears the marker and the resource reappears; when documented, an include-deleted filter surfaces the tombstone with its marker.
- *Negative path:* update (PUT) on an already-soft-deleted resource is rejected (404, also-accept 409/410) with the tombstone state unchanged; delete-missing → 404; double-delete stays consistent (no error escalation, idempotent tombstone).
- *Boundary:* default listing immediately after soft-delete (deleted id absent) vs include-deleted listing (present with marker); single vs repeated (double) delete on one id; row-count consistency after unique-key reuse.
- *Edge:* update-on-deleted must never silently apply; unique-key reuse after soft-delete must match the documented policy exactly — 201 if the tombstone frees the unique index, or 409 if the soft-deleted row still reserves the key (the classic soft-delete unique-constraint bug — either a wrong 409 or a duplicate active row / data corruption); cascade — soft-deleting a parent tombstones every documented child with no orphaned active child; every effect proven black-box by follow-up GET / listing, never by reading a DB row or log.

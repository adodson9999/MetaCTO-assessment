## ADDENDUM (v2 — exhaustive test-case + reporting standard)

LANE-SPECIFIC exhaustive coverage checklist (test-event-driven-api-triggers — owns broker/topic message semantics: broker consumption / topic-scoped ordering / DLQ / duplicate idempotency / schema-versioning):
- Happy: a well-formed event drives the resource to its documented state within the poll window (proven by black-box read-back); a next-compatible schema-version event still drives state per the documented compatibility policy.
- Negative: a malformed event is ERROR-logged + dead-lettered + state-unchanged + consumer stays healthy; an incompatible/unknown schema-version event is rejected + dead-lettered without applying state and without crashing the consumer.
- Boundary: a poison message is retried EXACTLY the documented count then dead-lettered (not retried beyond count); a burst exceeding steady throughput is fully drained to the correct terminal state within the documented lag budget (zero dropped).
- Edge: a duplicate event carrying the same broker idempotency key is applied exactly once (no double side effect); an out-of-order pair is resolved by the documented ordering/versioning rule (later/higher-version wins or stale dropped per policy).
- Broad / combinatorial: ordering-key/partition scope — two events on the SAME key applied in published order, events on DIFFERENT keys may interleave and each key reaches its own terminal state (guarantee-scope, distinct from version-based out-of-order resolution); each schema version × compatible-drives / incompatible-DLQ; consumer stays healthy across malformed × poison × burst conditions. Cite sibling owners for adjacent concerns: outbound HTTP-callback delivery, callback HMAC/tamper, webhook retry-backoff-DLQ over HTTP, webhook-target SSRF/allowlist → test-webhook-delivery (broker/topic DLQ of poison/malformed messages stays here); long-poll 204/no-event, hold-timeout, Last-Event-ID resume, concurrent pollers, connection-drop → test-long-polling-support; sequential Idempotency-Key replay on a synchronous HTTP endpoint → test-idempotency-of-endpoints (broker-key consumer dedupe stays here); parallel HTTP write races → test-concurrent-request-handling.

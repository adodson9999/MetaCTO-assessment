## Research basis

- draft-ietf-httpapi-ratelimit-headers-11 (May 2026): defines RateLimit-Limit/Remaining/Reset and the RateLimit / RateLimit-Policy dictionary fields; server MAY alter Reset between requests and clients MUST NOT assume full restoration at Reset (accommodates sliding-window) — grounds the header-math strengthening and the fixed-vs-sliding window-model case.
- Rate-limiting algorithm literature (Arcjet, API7, System Overflow, HLD Handbook): fixed-window boundary amplification (2N in 2s at the boundary), token-bucket burst-vs-sustained separation, sliding-window ~6% drift, cost/GCRA weighting — grounds `boundary_burst_amplification`, `burst_vs_sustained_token_bucket`, `cost_weighted_debit`, `window_reset_model_fixed_vs_sliding`.
- Distributed rate-limiter literature (arXiv 2602.11741, Redis rate-limiting howtos, "Five Problems That Break Your Counters"): INCR/Lua atomicity vs GET+SET races, AP eventual-consistency over-count/under-count, consistent-hash per-key sharding — grounds `distributed_counter_consistency`.
- OWASP "IP Spoofing via HTTP Headers" + Litestar/Mastodon/Rails/Acronis advisories: rotating X-Forwarded-For / forwarding headers resets per-IP counters and bypasses limits — grounds `limit_bypass_via_header_spoof` and `per_ip_isolation`.

## Gap summary

Original 7 cases covered a single at-limit burst, one over-limit request, wall-clock window probes, per-key isolation, per-endpoint-vs-global scope, and one header-decrement walk. Gaps closed: (1) window MODEL — original probes assumed a resettable window without distinguishing fixed vs sliding or testing boundary amplification; (2) isolation dimensions beyond per-key — per-ip and per-user were untested; (3) non-unit accounting — cost-weighted debits and token-bucket burst-vs-sustained shaping were absent; (4) counter integrity under concurrency/distribution — sequential-only bursts never exercised the lost-increment race; (5) the header-spoof bypass, the single most-reported real-world rate-limit defect, was untested. Header-math was hardened to the current IETF draft (non-negative Remaining, non-past Reset, byte-for-byte names).

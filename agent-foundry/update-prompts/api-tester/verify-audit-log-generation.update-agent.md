## Research basis
- Audit-trail completeness: every state-changing and security-relevant event (create/read/update/delete, login/logout, failed login, denied access) must emit exactly one tamper-evident entry — a silent (missing) entry is a coverage defect (OWASP A09:2021 Security Logging & Monitoring Failures).
- Entry fidelity: actor identity (real user, not service/anonymous), true source IP (not proxy/loopback), and accurate UTC/ISO-8601 timestamp within tolerance are the non-forgeable core of an audit record.
- Ordering & completeness: audit queries must partition the actions performed (no under/over-count, monotonic timestamps) — mirrors the contract-oracle pagination "partition" and list-sort "monotonic" invariants applied to the log surface.
- Immutability & tamper-evidence: audit entries are append-only; both delete AND modify attempts must be rejected (403/405) and read-back must prove the entry unchanged.
- Log injection (CRLF/newline/control-char/ANSI) — OWASP Log Injection: user-controlled text must be neutralized so it cannot forge or split a log line (CWE-117).
- Sensitive-value redaction — secrets/PII must never be written in cleartext to an audit log (OWASP A09 / logging cheat-sheet; CWE-532 Insertion of Sensitive Information into Log File).
- Contract-oracle §4 black-box read-back anchors every fidelity assertion to the audit-read surface, not to DB rows or injected instrumentation.

### Lane-specific exhaustive coverage checklist (in-lane, MECE across agents)

- **Happy path**: create/update(PUT)/patch(partial UPDATE)/delete each emits exactly one entry in the window; read-audit records a read event; login and logout each audited; before/after values captured on update; CREATE entry actor `user_id` + `ip_address` + `timestamp` + `resource_id` all correct (audit-entry fidelity).

- **Boundary**: timestamp exactly at the edge of `timestamp_tolerance_seconds`; ordered create→update→delete returns exactly three entries (no under/over-count) in monotonic non-decreasing timestamp order; window edge (entries just inside vs just outside the query window).

- **Broad / combinatorial**: each audited action type (CREATE/READ/UPDATE/PATCH/DELETE/LOGIN/LOGOUT/LOGIN_FAILED/READ_DENIED) × exactly-one-entry coverage; each required field (user_id, action_type, resource_id, timestamp, ip_address) × fidelity assertion; each tamper vector (delete, modify) × rejection + unchanged read-back.

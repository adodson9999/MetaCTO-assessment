## Research basis
- RFC 9745 (The Deprecation HTTP Response Header Field, 2025): Deprecation is a structured-field date indicating when deprecation takes effect; treated as a hint. Source: rfc-editor.org/rfc/rfc9745.html.
- RFC 8594 (The Sunset HTTP Header Field): Sunset is an HTTP-date for when the resource stops working; its timestamp MUST NOT be earlier than the Deprecation date; migration signalled via `Link rel="successor-version"` (RFC 8288). Sources: rfc-editor.org/info/rfc9745; zalando/restful-api-guidelines deprecation chapter.
- contract-oracle.md "Status semantics" (documented code exactly), "Documented capability" (documented-but-404/ignored → missing_capability), and "Validation" (never 5xx on a conflict) rows.

**Lane-specific exhaustive coverage checklist (ASPECT = API versioning across path/media-type/query channels; generic format negotiation and bare status codes are siblings').**
- Happy: a current version (path, media-type, and query-param when documented) serves a valid body against its schema and carries NO Deprecation header; the default-version behaves as documented when no version is supplied.
- Negative: an unsupported numeric path version returns 404; an unsupported non-numeric path version returns 400; an unsupported media-type / query-param version is rejected (404/406/400) — never a silent downgrade.
- Boundary: an out-of-range vendor version (e.g. v99, v0) is unsupported; the exact highest-supported vs first-unsupported version edge; Sunset is a valid HTTP-date NOT earlier than the Deprecation date.
- Edge: a deprecated version STILL serves a schema-valid 2xx body during its Sunset window (not 410/error); Deprecation parses as a structured-field date, Sunset is present, Link carries rel="successor-version"; an unsupported version is NOT silently downgraded to current (the current-only schema-diff field must be absent in any served body).
- Broad: per-version schema DIVERGENCE both directions — the current schema has the diff field, the deprecated schema omits it; ambiguous multi-channel version (path vs conflicting Accept vendor version) resolves deterministically by documented precedence or 400/406, never 5xx; each channel × version_status enumerated as its own case.
- Sibling owners for adjacent concerns: generic Accept/Content-Type/charset/encoding FORMAT negotiation and generic 406/415 → verify-content-type-negotiation; the bare 404/400/410 STATUS-CODE-value conformance divorced from a version selection → verify-response-status-codes; Deprecation/Sunset as CACHING freshness → verify-caching-headers; generic (non-per-version) response-body schema → validate-json-schema-responses.

Coverage exhaustive in-lane, MECE across agents — no duplicate cases.

# Update Spec — validate-json-schema-responses

## User prompt
Expand this agent's lane to the complete response-BODY schema-conformance universe it solely owns — strict ajv v8 validation of EVERY documented response code's body against its documented schema, covering additionalProperties (no undocumented/leaked fields), required-present-and-correctly-typed, nullability, format assertions, array-item + non-empty validation, enum-in-response membership, numeric/date-format assertions, envelope/pagination-wrapper consistency, and application/json content-type — while keeping it a pure schema-conformance bug-finder that emits ONE JSON object (never asserting an actual returned body/status/count/verdict — the deterministic harness runs ajv and records `observed`), feature-agnostic (byte-for-byte echo the runtime-supplied operationId, method, path, and documented status keys; otherwise refer to inputs only by role: endpoint_under_test; never assume/hardcode/name/mention any URL/path/host/resource/feature beyond that echo), with `expected_by_contract` taken ONLY from `references/contract-oracle.md` (the "Read", "Status semantics", and "Headers" rows — body matches the documented schema; Content-Type correct), and preserving all existing invariants below.

Mirror this agent's own golden.json shape EXACTLY. Top-level keys and nothing else: `agent`, `lane`, `descriptors`, `response_schema_map`, `validation_flags`, `out_of_scope` (null when in-lane), `baseline` (`{ "metric": "response_schema_fidelity", "value": 1.0 }`). Each `descriptors` entry has `role` (stable label such as `descriptor_2xx`, `descriptor_400`, `descriptor_404`, `descriptor_500`, `descriptor_201`, `descriptor_204`, `descriptor_409`, `descriptor_422`), `endpoint_role` (always `endpoint_under_test`), `method` (runtime method echoed unchanged), `expected_class` (the documented status key echoed unchanged), `also_accept` (array), an optional `recipe` object ONLY on the success descriptor (`{ "kind": "valid_request", "auth": "valid"|"none", "body": "known_valid_example_body"|null }`), and a maximally granular `steps` array logging build → send → validate. Each `response_schema_map` entry has `code` (status key echoed as a string), `has_json_schema` (boolean copied unchanged from input, no guessing), and `is_list` (boolean copied unchanged from input).

KEEP the existing 4 descriptors and their schema-map entries unchanged (descriptor_2xx with the valid_request recipe + list schema, descriptor_400, descriptor_404, descriptor_500), and KEEP the descriptors-array rule: emit exactly ONE descriptor per DOCUMENTED response code the runtime supplies — no more, no less. The additions below are the schema-assertion RECIPES the harness applies and the closed vocabulary of validation flags; they do NOT force undocumented codes to appear (a code is only tested if the runtime response-schema map documents it).

EXPAND the `validation_flags` object (still pinned exactly, never relaxed or invented). Existing flags retained: `additional_properties` = false, `required_present_and_typed` = true, `list_item_validation` = true, `content_type` = "application/json". ADD these NEW strict flags:
- `nullable_honored` = true — a field documented nullable accepts null; a field NOT documented nullable rejects null (ajv nullable / type-union checked explicitly, since ajv does not extend an enum/const with null automatically).
- `format_assertion` = true — ajv-formats registered and run in assertive (not advisory) mode so documented formats (email, uri, uuid, date, date-time, ipv4/ipv6) are enforced, not silently passed.
- `enum_in_response` = true — a documented enum field in the response body must hold only a documented member value.
- `numeric_and_date_format` = true — documented numeric constraints (type integer vs number, multipleOf, min/max) and date/date-time string formats in the response body are enforced.
- `envelope_consistency` = true — when responses use a documented wrapper/envelope (e.g. a data + meta/pagination shape), every code's body conforms to that same documented envelope, and a documented pagination wrapper's required meta fields (such as total/limit/offset or a cursor/Link marker) are present and typed.
- `unevaluated_properties` = false — for composed schemas (allOf/oneOf/anyOf) apply JSON-Schema-2020-12 `unevaluatedProperties:false` so an undocumented field cannot slip through a sibling subschema that additionalProperties:false alone would miss.
- `content_type_problem_json` = "application/problem+json" — accepted as the documented Content-Type ONLY for error-class response bodies whose schema is documented as a problem-details shape; the success class stays application/json.

EXTEND every descriptor's `steps` array to log the new assertions so the harness runs them, adding after the existing assert step: "assert documented-nullable fields accept null and non-nullable fields reject null", "assert registered ajv-formats validate documented email/uri/uuid/date/date-time formats assertively", "assert every documented enum field in the response holds only a documented member", "assert documented numeric type/range and date formats", and (where the response is documented as enveloped or a list/pagination wrapper) "assert the documented envelope/pagination wrapper shape and its required meta fields", and "for composed allOf/oneOf schemas assert unevaluatedProperties:false". The success (`descriptor_2xx`) list case additionally keeps "validate every list item against the item schema and assert the list is non-empty".

New closed vocabulary is the `validation_flags` KEYS above (existing four PLUS nullable_honored, format_assertion, enum_in_response, numeric_and_date_format, envelope_consistency, unevaluated_properties, content_type_problem_json); no flag key outside this closed set may ever be emitted, and no flag may be relaxed from the pinned value. The success-descriptor recipe KIND stays the single closed kind `valid_request`.

REMOVE / never emit (route to sibling owner, cite it): the status CODE-VALUE correctness itself — whether the code returned is 201 vs 200, 404 vs 410, the presence of Location on 201/3xx or Allow on 405 (owned by api-tester-verify-response-status-codes — this agent validates the BODY under whatever documented code, not that the code is right); the error-MESSAGE wording, human-readability, machine-code-field presence, envelope-clarity phrasing, and internal-detail-LEAK (stack/SQL/path/version/hostname) checks on a failure body (owned by api-tester-verify-error-message-clarity — this agent asserts additionalProperties:false structurally rejects undocumented fields, but the semantic "is this a stack-trace leak" judgment is the sibling's); request-payload/request-body schema validation (owned by api-tester-validate-request-payloads — this agent validates RESPONSE bodies only); the correlation-ID echo/propagation semantics (owned by api-tester-validate-correlation-id-propagation). On out-of-lane input emit a single out-of-lane error sentinel naming the owning sibling in `out_of_scope` and nothing else.

PRESERVE all invariants: emit exactly ONE JSON object with exactly the listed top-level keys and nothing else (no prose, no extra/renamed keys); never assert or guess an actual returned body, status, error count, field count, or conformance verdict (a separate deterministic harness sends each request to the one local target, runs ajv v8 against any documented response schema, and records the real outcome); one descriptor per DOCUMENTED response code (the success code and each documented 4xx/5xx), driven only by the runtime response-schema map with has_json_schema/is_list copied unchanged and never guessed; the success descriptor carries the valid_request recipe (auth `valid` when the endpoint requires authentication else `none`; body copied unchanged for POST/PUT/PATCH else null); feature-agnostic with byte-for-byte echo and fail-closed out-of-scope when no surface is provided; `expected_by_contract` sourced ONLY from contract-oracle.md; never carry an `also_accept` that swallows a standard code; confine all file access to FORGE_WORKSPACE; comply with Articles G1–G11; retain the self-awareness/code-review clause (all produced code is reviewed by every agent in agents/code-review/ and must score ≥85, looping until it does).

New total: descriptor count unchanged in structure (one per documented code — the golden fixture shows 4; a richer surface yields more, always one-per-documented-code); `validation_flags` grows from 4 to 11 pinned strict flags; each descriptor's `steps` array gains the 5–6 new assertion lines above.

## ADDENDUM (v2 — exhaustive test-case + reporting standard)
When running update-agent for this agent, pass the Change prompt above AND this ADDENDUM together as a single change prompt.

**No-verdict role.** This agent is now a pure, exhaustive test-case generator in its lane. It authors every case and fills the Expected Result (the definition of correct behavior, sourced from the contract oracle and the given spec). It does NOT execute, does NOT judge, and emits NO deviations, verdicts, or pass/fail. For every case it sets `actual_result` = "TO BE FILLED DURING EXECUTION" and `status` = `Not Executed`; a separate judge agent later executes the case, fills the actual result, and decides whether it is a bug. This section is governed by `00-AUTHORING-STANDARD-exhaustive-testcases.md`.

**Test Case ID prefix:** `TC-SCHEMA-NNN` (zero-padded, sequential, stable across runs).

**Render every case in the human-readable schema.** In addition to the machine descriptors/schema-map above, emit each test case with ALL of these human fields, in plain language, maximum detail: `test_case_id`, `title`, `description`, `category` (`happy`|`negative`|`boundary`|`edge`|`broad`), `feature_under_test`, `preconditions`, `test_data`, `test_steps`, `expected_result`, `actual_result` (="TO BE FILLED DURING EXECUTION"), `status` (=`Not Executed`), `postconditions`, `severity_hint`, `references`, `tags`. Keep this agent's existing machine fields (`role`, `endpoint_role`, `method`, `expected_class`, `also_accept`, `recipe`, `steps`, plus the `validation_flags` and `response_schema_map` entries) under a `machine` key on each case. Emit ONE JSON object with a `test_cases[]` array carrying every case (one per documented response code).

**Lane-specific exhaustive coverage checklist (ASPECT = response BODY schema/typing/additionalProperties/nullability/format/enum/envelope; NOT the code value and NOT the message wording).**
- Happy: the documented 2xx body validates under strict ajv v8 (required-present-and-correctly-typed); every list item validates against the item schema and the list is non-empty; Content-Type is application/json; a documented envelope/pagination wrapper carries its required meta fields (total/limit/offset or cursor/Link) present and typed.
- Negative: a body with an undocumented/extra field is rejected by additionalProperties:false; a documented-non-nullable field carrying null is rejected; a wrong-typed required field (string-for-integer, etc.) is rejected; a documented enum field holding an undocumented member is rejected.
- Boundary: numeric constraints (integer vs number, multipleOf, min/max) enforced at the edges; empty list vs single-item vs many-item body against the item schema; the exact additionalProperties boundary where one undocumented key appears.
- Edge: documented-nullable fields accept null while non-nullable reject it; assertive ajv-formats (email/uri/uuid/date/date-time/ipv4/ipv6) reject malformed values instead of silently passing; composed allOf/oneOf/anyOf schemas apply unevaluatedProperties:false so an undocumented field cannot slip through a sibling subschema; a documented error body validates against application/problem+json for error classes while success stays application/json.
- Broad: one descriptor per DOCUMENTED response code (success + each documented 4xx/5xx), each validating its documented schema; every pinned validation flag (additional_properties, required_present_and_typed, list_item_validation, content_type, nullable_honored, format_assertion, enum_in_response, numeric_and_date_format, envelope_consistency, unevaluated_properties, content_type_problem_json) exercised as a distinct assertion step.
- Sibling owners for adjacent concerns: status CODE-value correctness (201 vs 200, 404 vs 410, Location/Allow) → verify-response-status-codes; error-MESSAGE wording + the semantic stack-trace/SQL/path/version/hostname LEAK judgment → verify-error-message-clarity (this agent only STRUCTURALLY rejects undocumented fields); REQUEST-body schema → validate-request-payloads; correlation-ID echo/propagation → validate-correlation-id-propagation.

Coverage exhaustive in-lane, MECE across agents — no duplicate cases.

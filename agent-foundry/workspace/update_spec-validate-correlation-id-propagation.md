**Lane-specific exhaustive coverage checklist (ASPECT = correlation-ID / request-ID only; generic header forwarding and trace-continuation are siblings').**
- Happy: an inbound correlation id is echoed byte-for-byte on the response; the same id appears in the API log and every downstream log unmodified; a provided valid id is honoured, not replaced by a server-generated one.
- Negative: a malformed correlation id (control chars / bad format) is rejected or sanitized, never reflected raw; a request with no correlation header gets a fresh UUIDv4 auto-generated and propagated.
- Boundary: an over-long-but-syntactically-valid id is bounded to the documented length or rejected, never reflected unbounded into any response header or log; the exact length-bound edge.
- Edge: two requests sent without a correlation header receive two DISTINCT auto-generated ids (uniqueness across requests); the X-Request-ID alias (when it is the runtime-supplied header role) carries the identical echo + propagation contract; an error-forcing request still echoes and logs the provided id.
- Broad: id-in-error-BODY-field (value equals the sent id byte-for-byte) distinct from id-in-response-HEADER; propagation verified by a COUNT assertion over the full downstream service list; each assertion flag enumerated as its own case.
- Sibling owners for adjacent concerns: generic header forwarding (Authorization/traceparent/tracestate/B3/X-Forwarded-*/custom) and hop-by-hop stripping → validate-header-propagation; CRLF/header-injection on a GENERIC custom header → validate-header-propagation (this agent covers CRLF only within the correlation-id VALUE); error-envelope SHAPE/wording/leak → verify-error-message-clarity (this agent asserts the id VALUE presence only); traceparent same-trace-id continuation math → validate-header-propagation.

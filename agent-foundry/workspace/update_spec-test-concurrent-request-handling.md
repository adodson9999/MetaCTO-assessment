## ADDENDUM (v2 — exhaustive test-case + reporting standard)
When running update-agent for this agent, pass the Change prompt above AND this ADDENDUM together as a single change prompt.

This agent is a **pure, exhaustive test-case generator** in its lane. It makes **NO bug judgement, verdict, or pass/fail call**. For every case it authors the scenario and fills the *Expected Result* (the definition of correct behavior, sourced from `references/contract-oracle.md` and the given feature spec). It leaves `actual_result` = `"TO BE FILLED DURING EXECUTION"` and `status` = `Not Executed`. It emits **no** deviations, no findings, no verdict, no `is_bug`, no pass/fail counts — a **separate judging/executor agent** runs the cases and decides whether an Actual Result is a bug. This role is governed by `00-AUTHORING-STANDARD-exhaustive-testcases.md`.

**Test Case ID prefix: `TC-CONC-NNN`** (zero-padded, sequential, stable across runs).

**Render every case in the human-readable schema.** In addition to the existing machine fields (keep `role`, `endpoint_role`, `method`, `recipe`, `concurrency`, `expected_class`, `also_accept`, `asserts`, the `steps` object-array, `expected_by_contract`, `expected_by_docs` intact under a `machine` key on each case, plus the top-level `plan`, `execution`, `log`, `report`, `vu_id_template_token` fields, and preserve `[VU_ID]` byte-for-byte), every test case MUST also carry these plain-language fields: `test_case_id`, `title`, `description`, `category` (`happy`|`negative`|`boundary`|`edge`|`broad`), `feature_under_test`, `preconditions`, `test_data`, `test_steps`, `expected_result`, `actual_result` (="TO BE FILLED DURING EXECUTION"), `status` (=`Not Executed`), `postconditions`, `severity_hint`, `references`, `tags`. Still output ONE JSON object with a `test_cases[]` array.

**Lane-specific exhaustive coverage checklist (parallel/simultaneous concurrency lane):**

- *Happy path:* N concurrent identical reads return byte-identical bodies; N concurrent unique writes commit with DB count-delta = concurrency (zero duplicates, zero missing).
- *Negative path:* concurrent writers to one resource yield exactly one winner and stale writers rejected 409/412 (no lost update); N concurrent creates on one unique key → exactly one 201 and the rest 409.
- *Boundary:* the exact winner-count under an optimistic-lock race (exactly one, no more, no fewer); exactly one DB row for one shared unique key or shared Idempotency-Key despite N simultaneous requests; zero 5xx anywhere across every case.
- *Edge:* lost-update anomaly — final persisted state must equal the winning write, never a stale overwrite; create+delete TOCTOU on a single id — final state equals exactly one legal serial ordering (created-then-deleted → absent, or deleted-then-created → present), never a torn/partial row; parallel same-Idempotency-Key race — exactly one side effect / one DB row despite simultaneity (the parallel counterpart the idempotency sibling hands off); thundering-herd sustained waves must not degrade to 5xx or partial responses.
- *Broad / combinatorial:* each race type (read, unique-write, optimistic-update, same-unique-key create, create+delete TOCTOU, same-key idempotency, sustained herd) × the sanctioned direct-DB read-back oracle; the zero-5xx invariant asserted on every case.

Direct DB count-delta / row-count is the sanctioned black-box oracle here (the one place a DB read is correct — for a race the HTTP surface cannot expose atomically); all other invariants read-back where observable. Cite sibling owners for adjacent concerns: sequential single-request replay (GET/PUT/DELETE/POST), same-key-different-body sequential conflict, first-response caching, key TTL/scoping → `api-tester-test-idempotency-of-endpoints` — this lane owns ONLY the *parallel* same-key race; the moment requests are sequential they belong to the idempotency sibling; the single ordered CRUD lifecycle and sequential If-Match/412/428 preconditions and sequential duplicate-409 → `api-tester-verify-crud-operation-integrity`; rate-limit counting / 429 under burst → `test-rate-limit-enforcement` (this lane asserts stability/no-5xx, not limit math).

Coverage exhaustive in-lane, MECE across agents — no duplicate cases.
